The northstar: complete TypeScript port of libmpfr (~600 public functions), pure Node stdlib + native BigInt, no npm deps. The port is generated by a ralph loop driving sonnet L3 across each function, root-to-leaves of the call graph.

**Why:** This is a warmup. The real targets are FLINT, then bigger projects. MPFR is chosen because it has (a) a well-defined public API with strong invariants (correct rounding, ternary flags), (b) a comprehensive existing test suite to mine for golden cases, and (c) is small enough to complete in days, not weeks.

**Constraints (locked):**
- Pure TS: Node stdlib + native BigInt only. No npm.
- API: idiomatic immutable (see [[decision-api-shape]]).
- Substrate: faithful port of GMP mpn_* + MPFR helpers first (see [[decision-substrate]]).
- Loop: sonnet L3 → auto-escalate to opus L3 once → park (see [[decision-failure-policy]]).
- Perf gate: moderate-to-strict with separate perf grade (see [[decision-perf-gate]]).

**Brutal harness requirements:**
- Per-function golden master with happy + edge + adversarial + fuzz cases
- Adapt MPFR's own `tests/*.c` files as additional goldens
- Grade rewards correctness AND optimized fast code
- Tiered visibility: which functions are correct-and-fast vs correct-but-slow vs broken

**Open architectural questions (not yet resolved):**
- State management: SQLite vs JSONL? (proposed: SQLite for queryability)
- Per-test isolation: worker_threads from day one to survive infinite loops
- Call graph extraction: clang AST dump vs nm-on-archive + grep
- Test suite mining: parse `tests/*.c` for `mpfr_set_str(); mpfr_X(); assert()` triples
- Pilot subset: ~7-10 fns exercising substrate end-to-end before scaling

How to apply: this is the entire active project. Every architectural choice should serve the ralph-loop-grinds-out-the-library workflow.
